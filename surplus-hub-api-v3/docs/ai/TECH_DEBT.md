# 기술 부채 및 개선 로드맵

Surplus Hub API v3의 AI 기능에 대한 알려진 기술 부채, 제한사항, 향후 개선 영역을 체계적으로 문서화합니다.

---

## 문서 개요

이 문서는 AI 기능의 기술 부채를 다음 관점에서 분류합니다:
- **코드 품질**: 구현 방식의 개선 여지
- **성능 최적화**: 응답 시간, 메모리 사용량 개선
- **확장성**: 대량 데이터 및 트래픽 대응
- **보안**: 취약점 및 위험 요소
- **운영성**: 모니터링, 로깅, 장애 대응

## 우선순위 범례
- 🔴 **P1 (High)**: 즉시 해결 필요 (보안/성능 이슈)
- 🟡 **P2 (Medium)**: 다음 스프린트 권장 (사용자 경험 영향)
- 🟢 **P3 (Low)**: 여유 있을 때 개선 (품질 향상)

## 1. 알려진 기술 부채

### ⚡ 성능 및 확장성 관련

#### TD-001: 임베딩 모델 메모리 사용량

- **우선순위**: 🔴 P1 (High)
- **설명**: bge-m3 모델이 약 2.4GB의 메모리를 사용합니다. 서버 스케일아웃 시 각 워커마다 모델이 로딩될 수 있어 메모리 부담이 급격히 증가할 수 있습니다.
- **현재 대응**: `gunicorn --preload` 옵션으로 워커 간 메모리 공유를 시도하고 있습니다.
- **개선 방안**:
  - 별도 임베딩 서비스로 분리 (gRPC/HTTP 기반)
  - ONNX Runtime으로 모델 경량화
  - 외부 임베딩 API (OpenAI Embeddings, Cohere Embed) 전환 고려
  - Serverless 환경에서 콜드 스타트 시간 측정 및 최적화
- **영향 범위**: `src/ai/embedding_service.py`
- **예상 공수**: 16시간 (중기 계획)

---

#### TD-002: embedding_hook 동기 처리로 인한 응답 지연

- **우선순위**: 🟡 P2 (Medium)
- **설명**: 자재 생성/수정 시 임베딩을 동기적으로 생성하여 응답 시간이 느려집니다 (약 500ms).
- **현재 구현**: `src/utils/embedding_hook.py`의 `after_insert`, `after_update` 이벤트 훅
- **문제점**: 대량 자재 등록 시 성능 저하
- **개선 방안**:
  - FastAPI BackgroundTasks로 비동기 처리
  - Celery + Redis 큐 기반 배치 처리
  - Dirty flag 기반 업데이트 (일정 주기마다 배치)
- **영향 범위**: `src/utils/embedding_hook.py`, `src/api/endpoints/materials.py`
- **예상 공수**: 4시간
- **효과**: 자재 등록 응답 시간 500ms → 50ms (10배 개선)

---

#### TD-003: 하이브리드 검색의 단순한 키워드 매칭

- **우선순위**: 🟡 P2 (Medium)
- **설명**: 현재 키워드 점수는 `ILIKE '%{keyword}%'` 기반의 0/1 이진값입니다. 정교한 키워드 스코어링이 없어 검색 품질이 제한적입니다.
- **현재 구현**: `src/ai/search/hybrid_search.py` 내 `_calculate_keyword_score()`
- **개선 방안**:
  - PostgreSQL Full-Text Search (tsvector/tsquery) 통합
  - 한국어 형태소 분석기 (mecab-ko, Kiwi) 활용
  - BM25 스코어링 알고리즘 도입
  - 동의어 사전 (synonym dictionary) 구축
- **영향 범위**: `src/ai/search/hybrid_search.py`, DB 마이그레이션
- **예상 공수**: 12시간

---

#### TD-004: 검색 결과 캐싱 미구현

- **우선순위**: 🟢 P3 (Low)
- **설명**: 동일한 검색 쿼리에 대해 매번 벡터 검색과 하이브리드 스코어링을 수행합니다.
- **영향**: 높은 트래픽 시 DB 부하 증가
- **개선 방안**:
  - Redis 기반 검색 결과 캐싱 (TTL: 5-10분)
  - 쿼리 정규화 (공백, 대소문자 통일)
  - 인기 검색어 프리페칭
- **영향 범위**: `src/ai/search/hybrid_search.py`, `docker-compose.yml` (Redis 추가)
- **예상 공수**: 6시간
- **효과**: 자주 검색되는 쿼리 응답 시간 50% 감소

#### TD-005: 벡터 검색 시 Null 임베딩 자재 제외

- **우선순위**: 🟢 P3 (Low)
- **설명**: `embedding_vector`가 `NULL`인 자재는 벡터 검색 결과에 포함되지 않습니다.
- **현재 대응**:
  - `src/utils/embedding_hook.py`로 자재 생성/수정 시 자동 임베딩 생성
  - `scripts/backfill_embeddings.py` 스크립트로 수동 백필
- **제한사항**: 대량 자재 생성 시 임베딩 생성 지연 가능
- **개선 방안**:
  - 비동기 임베딩 생성 큐 (Celery + Redis)
  - 임베딩 생성 우선순위 설정 (인기 자재 우선)
  - 임베딩 생성 실패 시 재시도 메커니즘
- **예상 공수**: 8시간 (TD-002와 함께 진행 시 효율적)

#### TD-006: Dockerfile torch CPU-only 최적화

- **우선순위**: 🟡 P2 (Medium)
- **설명**: 기본 torch 설치 시 GPU 드라이버 포함으로 불필요한 용량 증가
- **영향 범위**: `Dockerfile`
- **효과**: 이미지 크기 약 1GB 감소 (2.5GB → 1.5GB)
- **예상 공수**: 1시간

현재 문제:
```dockerfile
# ❌ 기본 torch 설치 (GPU 드라이버 포함, 불필요한 용량)
RUN pip install sentence-transformers
```

해결 방법:
```dockerfile
# ✅ CPU-only torch 명시적 설치
RUN pip install torch --index-url https://download.pytorch.org/whl/cpu
RUN pip install sentence-transformers
```

#### TD-010: Rate Limiting이 IP 기반

- **우선순위**: 🟢 P3 (Low)
- **설명**: `slowapi`의 기본 rate limiting이 IP 기반입니다. 동일 IP의 다른 사용자가 영향을 받을 수 있습니다 (예: 회사 NAT, 공용 Wi-Fi).
- **현재 설정**: `@limiter.limit("10/minute")`
- **개선 방안**:
  - 인증된 사용자 ID 기반 rate limiting
  - IP + User ID 하이브리드 방식
  - Redis 기반 분산 rate limiting
- **영향 범위**: `src/core/rate_limit.py`, 모든 AI 엔드포인트
- **예상 공수**: 3시간

#### TD-012: Pydantic v2 Config 마이그레이션

- **우선순위**: 🟢 P3 (Low)
- **설명**: Pydantic v1 스타일의 `Config` 클래스 사용으로 deprecation 경고 발생
- **영향 범위**: `src/schemas/ai_schemas.py` (7개 스키마)
- **예상 공수**: 2시간

현재 문제:
```python
# ❌ Deprecated (Pydantic v1 스타일)
class ImageAnalysisResponse(BaseModel):
    class Config:
        from_attributes = True
        populate_by_name = True
```

해결 방법:
```python
# ✅ Pydantic v2 스타일
from pydantic import ConfigDict

class ImageAnalysisResponse(BaseModel):
    model_config = ConfigDict(
        from_attributes=True,
        populate_by_name=True
    )
```

## 2. 성능 개선 필요 영역

### 2.1 대량 데이터 시나리오

| 시나리오 | 현재 성능 | 목표 | 개선 방안 |
|---------|----------|------|----------|
| 자재 수 10만+ | 미검증 | 검색 < 200ms | HNSW 파라미터 튜닝 (ef_construction, M) |
| 동시 검색 100+ | 미검증 | 처리율 > 50 req/s | 연결 풀 최적화, 캐싱 (TD-004) |
| 임베딩 생성 1000+ | 배치 처리 시간 미측정 | < 5분 | GPU 활용, 병렬 처리 |

### 2.2 임베딩 모델 최적화

- **현재 문제**: 모델 로드 시간 ~3-5초 (콜드 스타트)
- **개선 방안**:
  - 모델 양자화 (INT8, FP16)
  - ONNX Runtime 전환
  - 모델 프리로딩 (서버 시작 시)

### 2.3 LLM API 응답 시간 변동

- **현재 상태**: Gemini/OpenAI API latency 0.5-3초 (불안정)
- **영향**: 가격 제안, 채팅 추천 기능의 응답 지연
- **개선 방안**:
  - 타임아웃 설정 및 폴백 전략
  - 응답 스트리밍 (가능한 경우)
  - 캐싱 (동일 요청 반복 시)

### 2.4 데이터베이스 쿼리 최적화

- **벡터 검색 쿼리**: `ORDER BY embedding_vector <=> $1 LIMIT 20`
  - HNSW 인덱스 존재하지만 파라미터 튜닝 미실시
  - 대량 데이터에서 성능 검증 필요
- **하이브리드 검색 쿼리**: 2단계 쿼리 (벡터 + 키워드)
  - 쿼리 통합 가능성 검토
  - Subquery 최적화
