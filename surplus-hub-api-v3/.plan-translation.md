# P3: 번역 기능 구현 계획서

## 1. 현재 AI 인프라 파악 (실제 파일 기반)

### AI 클라이언트 구조
- **Gemini** (`app/ai/clients/gemini.py`): `gemini-2.5-flash-lite` 모델. 이미지 분석 전용. Thread-safe singleton + tenacity retry (3회, exponential backoff). 타임아웃 30초.
- **OpenAI** (`app/ai/clients/openai_client.py`): `gpt-5-nano` (기본), `gpt-5-mini` (추론용). 텍스트 생성 전용. Thread-safe singleton, 내장 retry 3회, 타임아웃 30초. `generate_text()` / `generate_text_with_history()` 두 함수 제공.

### 서비스 레이어 패턴
- `app/ai/services/registration.py`: Gemini(이미지) + OpenAI(텍스트) 조합. 결과를 Pydantic 스키마로 래핑.
- `app/ai/services/qa_bot.py`: OpenAI만 사용. DB에서 데이터 조회 후 프롬프트 구성 -> `generate_text()` 호출 -> JSON 파싱 -> Pydantic 응답.
- 모든 서비스는 동기 함수.

### Rate Limiting
- `slowapi.Limiter` 사용. Redis URL이 있으면 Redis 기반, 없으면 인메모리.
- 엔드포인트별 제한: 10/min (이미지, 설명, 가격), 20/min (채팅), 5/min (커뮤니티).

### Redis
- `app/core/redis.py`: `redis.asyncio`로 async Redis 클라이언트 제공. `REDIS_URL` 설정 의존.
- 현재 캐싱에 사용되고 있지 않음 — rate limiter에서만 간접 사용.

### 기존 스키마 패턴
- `app/schemas/ai_schemas.py`: Pydantic v2 BaseModel + `Config(populate_by_name=True)` + camelCase alias.

## 2. 번역 기능 설계

### 2.1 번역 대상
**자재 설명 + 채팅 메시지 모두 지원** (범용 텍스트 번역 API로 설계)
- 건설 잉여자재 마켓플레이스의 특성상 외국인 노동자/바이어가 주요 타겟
- 자재 설명(description), 채팅 메시지, 커뮤니티 글 등 모든 텍스트에 범용 적용 가능

### 2.2 API 설계

**엔드포인트**: `POST /api/v1/ai/translate`

**Request 스키마** (`TranslateRequest`):
```python
class TranslateRequest(BaseModel):
    text: str = Field(..., min_length=1, max_length=5000)
    source_language: str = Field("auto", alias="sourceLanguage")  # "auto", "ko", "en", "zh", "vi", "th" 등
    target_language: str = Field(..., alias="targetLanguage")      # "ko", "en", "zh", "vi", "th" 등
```

**Response 스키마** (`TranslateResponse`):
```python
class TranslateResponse(BaseModel):
    translated_text: str = Field(alias="translatedText")
    source_language: str = Field(alias="sourceLanguage")  # 감지된 또는 지정된 원본 언어
    target_language: str = Field(alias="targetLanguage")
    model_used: str = Field(alias="modelUsed")
```

**Rate Limit**: `20/minute` (채팅 수준 빈도 허용)

### 2.3 AI 모델 선택: OpenAI `gpt-5-nano`

**근거**:
1. 번역은 텍스트→텍스트 작업이므로 Gemini(이미지 특화)보다 OpenAI 텍스트 클라이언트가 적합
2. `gpt-5-nano`는 이미 `generate_text()`로 검증된 인프라가 존재 — 신규 클라이언트 불필요
3. 번역은 추론 복잡도가 낮으므로 `gpt-5-nano`로 충분 (비용/속도 최적)
4. 기존 openai_client.py에 내장 retry + 타임아웃이 이미 구현됨

### 2.4 서비스 레이어

**새 파일**: `app/ai/services/translation.py`
**새 프롬프트**: `app/ai/prompts/translation.py`

```python
# app/ai/services/translation.py
def translate_text(text: str, source_language: str, target_language: str) -> TranslateResponse:
    """번역 수행. Redis 캐싱 적용."""
    # 1. 캐시 키 생성 (text hash + source + target)
    # 2. Redis 캐시 확인 (hit시 즉시 반환)
    # 3. OpenAI generate_text() 호출
    # 4. 결과 Redis 캐시에 저장 (TTL 1시간)
    # 5. TranslateResponse 반환
```

### 2.5 캐싱 전략

- **캐시 키**: `translate:{sha256(text)[:16]}:{source}:{target}`
- **TTL**: 3600초 (1시간) — 번역 결과는 동일 입력에 대해 변하지 않으므로
- **Redis 미설정 시**: 캐싱 스킵, 매번 AI API 호출 (기존 패턴과 동일하게 graceful degradation)
- **동기 Redis 클라이언트 필요**: 기존 `redis.asyncio`는 async 전용. 서비스 레이어가 동기이므로 `redis.Redis` 동기 클라이언트를 별도로 사용하거나, `hashlib` 기반 인메모리 LRU 캐시 사용.
  - **결정**: 인메모리 `functools.lru_cache`는 프로세스 간 공유 불가하므로, 동기 `redis.Redis` 클라이언트를 translation 서비스 내에서 직접 생성. Redis 미설정 시 스킵.

## 3. 동시성 테스트 시나리오

### 시나리오 A: 100개 동시 번역 요청 시 API rate limit 처리 방안

**방안**:
- slowapi `@limiter.limit("20/minute")` — IP당 분당 20회 초과 시 429 반환
- OpenAI API 자체 rate limit 초과 시 → `_is_rate_limit_error()` 감지 → HTTP 429 + `Retry-After: 60` 헤더
- OpenAI 클라이언트 내장 `max_retries=3` 자동 재시도

```python
# test_translate_rate_limit.py (초안)
class TestTranslateRateLimit:
    """POST /api/v1/ai/translate — rate limit scenarios"""

    @patch("app.ai.services.translation.translate_text")
    def test_rate_limit_exceeded_returns_429(
        self, mock_translate, client: TestClient, auth_headers: dict
    ):
        """20/min 초과 시 429 반환 확인"""
        mock_translate.return_value = TranslateResponse(
            translatedText="Hello",
            sourceLanguage="ko",
            targetLanguage="en",
            modelUsed="gpt-5-nano",
        )
        # 21번 연속 호출
        for i in range(21):
            resp = client.post(
                "/api/v1/ai/translate",
                json={"text": f"안녕하세요 {i}", "targetLanguage": "en"},
                headers=auth_headers,
            )
            if resp.status_code == 429:
                assert "Retry-After" in resp.headers or resp.status_code == 429
                return
        # slowapi 인메모리 모드에서는 정확한 limit 적용이 다를 수 있음
        # 따라서 429가 한 번이라도 발생하면 통과

    @patch("app.ai.clients.openai_client.generate_text")
    def test_upstream_rate_limit_returns_429(
        self, mock_gen, client: TestClient, auth_headers: dict
    ):
        """OpenAI API 429 에러 시 클라이언트에도 429 전파 확인"""
        error = Exception("Rate limit exceeded (429)")
        error.status_code = 429
        mock_gen.side_effect = error

        resp = client.post(
            "/api/v1/ai/translate",
            json={"text": "안녕하세요", "targetLanguage": "en"},
            headers=auth_headers,
        )
        assert resp.status_code == 429
        assert "rate limit" in resp.json()["detail"].lower()
```

### 시나리오 B: 동일 텍스트 동시 요청 시 중복 AI API 호출 방지 (캐싱 전략)

**방안**:
- Redis SET + GET으로 결과 캐싱 (동일 text+source+target 조합)
- 동일 텍스트 두 번째 요청부터는 캐시 히트 → AI API 호출 없음
- 진정한 동시 요청(두 요청이 동시에 캐시 미스):
  - Redis SETNX 기반 "번역 진행 중" 락은 복잡도 대비 이득이 적음 (번역 소요 ~1초, 중복 호출 비용 미미)
  - **결정**: 단순 cache-aside 패턴 채택. 2번째 이후 요청은 캐시 히트.

```python
# test_translate_caching.py (초안)
class TestTranslateCaching:
    """번역 캐싱 동작 검증"""

    @patch("app.ai.services.translation._get_sync_redis")
    @patch("app.ai.clients.openai_client.generate_text")
    def test_cached_translation_skips_ai_call(
        self, mock_gen, mock_redis, client: TestClient, auth_headers: dict
    ):
        """동일 텍스트 두 번째 요청 시 AI API 미호출 확인"""
        mock_redis_instance = MagicMock()
        mock_redis.return_value = mock_redis_instance

        # 1차: 캐시 미스 -> AI 호출
        mock_redis_instance.get.return_value = None
        mock_gen.return_value = '{"translated_text": "Hello", "detected_language": "ko"}'

        resp1 = client.post(
            "/api/v1/ai/translate",
            json={"text": "안녕하세요", "targetLanguage": "en"},
            headers=auth_headers,
        )
        assert resp1.status_code == 200
        assert mock_gen.call_count == 1

        # 2차: 캐시 히트 -> AI 미호출
        mock_redis_instance.get.return_value = '{"translated_text": "Hello", "detected_language": "ko"}'

        resp2 = client.post(
            "/api/v1/ai/translate",
            json={"text": "안녕하세요", "targetLanguage": "en"},
            headers=auth_headers,
        )
        assert resp2.status_code == 200
        assert mock_gen.call_count == 1  # 여전히 1회

    @patch("app.ai.services.translation._get_sync_redis")
    @patch("app.ai.clients.openai_client.generate_text")
    def test_translation_works_without_redis(
        self, mock_gen, mock_redis, client: TestClient, auth_headers: dict
    ):
        """Redis 없을 때도 번역 정상 동작 확인"""
        mock_redis.return_value = None  # Redis 비활성
        mock_gen.return_value = '{"translated_text": "Hello", "detected_language": "ko"}'

        resp = client.post(
            "/api/v1/ai/translate",
            json={"text": "안녕하세요", "targetLanguage": "en"},
            headers=auth_headers,
        )
        assert resp.status_code == 200
        assert mock_gen.call_count == 1
```

## 4. 가정 목록 (실제 파일 확인 후에도 미확인)

1. **지원 언어 범위**: "ko", "en", "zh", "vi", "th", "ja" 등 — 사용자에게 확인 필요하지만, GPT 모델이 다국어를 지원하므로 제한 없이 열어두고 프롬프트에서 처리
2. **Redis 동기 클라이언트**: `redis` 패키지가 이미 설치되어 있음 (`redis.asyncio` 사용 중이므로). 동기 `redis.Redis`도 같은 패키지에서 import 가능.
3. **인증 필수**: 기존 AI 엔드포인트 패턴상 `get_current_active_user` 의존성 사용 — 번역도 동일 적용
4. **max_length 5000**: 건설 자재 설명/채팅 메시지 기준 충분한 길이. GPT-5-nano 토큰 한도 내.
